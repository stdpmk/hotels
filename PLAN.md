# Архитектура сервиса бронирования отелей

Высокоуровневое описание архитектуры. Используется **Layered Architecture** — чёткое разделение ответственности между слоями.

---

## Слои приложения

```
┌─────────────────────────────────────┐
│            HTTP Layer               │  ← маршруты, хендлеры, middleware
├─────────────────────────────────────┤
│           Service Layer             │  ← бизнес-логика
├─────────────────────────────────────┤
│              DB Layer               │  ← запросы к PostgreSQL
├─────────────────────────────────────┤
│           Models Layer              │  ← типы данных, общие для всех слоёв
└─────────────────────────────────────┘
```

### models
Все типы данных приложения: `Hotel`, `Room`, `Booking`, `User`.  
Используются во всех слоях — не зависят ни от чего.

### db
Слой работы с базой данных.  
Реализует интерфейсы репозиториев: `HotelRepository`, `RoomRepository`, `BookingRepository`, `UserRepository`.  
Только SQL — никакой бизнес-логики.

### services
Бизнес-логика: проверка доступности номеров, расчёт стоимости, правила отмены брони.  
Зависит от интерфейсов `db`-слоя — не от конкретных реализаций.

### http
REST API: маршруты (Gorilla Mux), хендлеры, middleware (JWT-аутентификация, логирование).  
Зависит только от `services`.

## Инфраструктурные компоненты

Сейчас не реализуются, но архитектура должна их предусматривать.

### Миграции (с первого дня)
Управление схемой БД через SQL-файлы с версионированием.  
Инструмент: **goose** или **golang-migrate**.  
```
migrations/
├── 001_create_users.sql
├── 002_create_hotels.sql
├── 003_create_rooms.sql
└── 004_create_bookings.sql
```

### Кеш (Redis) — будущее
Кеширование часто запрашиваемых данных: список отелей, детали номеров.  
Встраивается между `services` и `db` без изменения интерфейсов.

### Очереди сообщений (будущее)
Асинхронные задачи: отправка email-подтверждений, уведомления об отмене брони.  
Варианты: **RabbitMQ**, **Kafka**.  
`services` публикует событие → воркер обрабатывает независимо.

### Конфигурация
Все параметры через переменные окружения (`.env`).  
Единый `config`-пакет читает их при старте.

## Структура директорий

```
2gis/
├── cmd/server/main.go       # точка входа
├── internal/
│   ├── config/              # чтение env
│   ├── models/              # типы данных
│   ├── db/                  # репозитории + интерфейсы
│   ├── services/            # бизнес-логика
│   └── http/
│       ├── handler/         # хендлеры
│       └── middleware/      # JWT, логирование
├── migrations/              # SQL-миграции
├── docs/                    # OpenAPI спецификация
└── README.md
```

---

## Принципы

- Слои зависят только от нижестоящих, никогда наоборот
- `db` и `services` работают через интерфейсы — легко заменить реализацию
- Инфраструктурные компоненты (кеш, очереди) подключаются без переписывания логики
